# Route-Map Fuzzy 优化分析报告

## 背景

route_engine.py 已支持 `fuzzy: true` 字段，两种匹配模式：

| 规则类型 | fuzzy 效果 | 实现 |
|---------|-----------|------|
| `phrase` | 中文子序列匹配 | pattern 的 CJK 字符按顺序出现在 input 中即可（如"写代码"→匹配"写一段代码"） |
| `keyword` | 汉字重叠率 ≥ 0.6 | 按字符集合重叠率计算（如"架构"→匹配"构架"，不匹配"框架"） |
| `regex` | 静默忽略 | regex 加 fuzzy 无意义 |

**核心原则**：
- 开启 fuzzy 的价值：用户可能用变体表达同一意图，精确匹配会漏掉
- 开启 fuzzy 的风险：变体可能偏离原始意图，导致误匹配到其他 Agent
- regex 规则不需要 fuzzy（regex 本身已覆盖变体）
- 负权重（误匹配防护）规则**不应**加 fuzzy —— 会削弱防护效果

### 1. programmer.yaml —— 代码编写与调试

#### 正面规则（正权重）

| # | 类型 | pattern | 权重 | fuzzy 建议 | 理由 |
|---|------|---------|------|-----------|------|
| 1 | phrase | 实现 | 0.8 | ❌ 不建议 | "实现"是高频泛用词。开启子序列后，"实现功能""实施方案""实施并展现"都会匹配，严重污染编码意图 |

| 3 | phrase | 编程 | 0.8 | ❌ 不建议 | 子序列"编"+"程"会匹配"编程语言""编程式""编制流程"，误匹配到非编码场景 |

## 特别风险提示

1. **`图`字规则**（ui-designer，weight 0.5）：绝对不要加 fuzzy，否则"架构图"、"流程图"、"图片"全部匹配，产生海量误匹配
2. **负权重规则加 fuzzy**：负权重本意是降低误匹配，如果加 fuzzy 会扩大负匹配范围导致本应路由到该 Agent 的输入被错误降权
3. **keyword fuzzy 只用于 overrides 的 `架构`**：当前只有一条 keyword 规则（pm-agent overrides 的"架构"），其他规则都是 phrase/regex 类型
4. **phrase + fuzzy 测试建议**：建议在测试环境中逐条验证子序列匹配，确认「什么输入会命中」在预期范围内
